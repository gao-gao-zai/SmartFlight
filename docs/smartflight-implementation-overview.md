# SmartFlight 各模块实现概览

## 1. 总体实现思路

SmartFlight 的核心不是单一页面，而是一套持续运行的自动化系统。建议拆成以下部分：

1. 接入与授权
2. 应用扫描与应用库
3. 前台应用监控
4. 屏幕状态监控
5. 网络环境监控
6. 规则引擎
7. 联网控制执行器
8. 保活与恢复
9. 日志与诊断
10. 设置与主题

整体数据流建议如下：

1. 系统事件或状态变化被监听到
2. 运行时服务收集当前上下文
3. 规则引擎根据配置作出决策
4. 执行器负责真正切换飞行模式或移动数据
5. 结果写入日志并刷新界面状态

## 2. 接入与授权

### 2.1 目标

- 确保用户具备至少一种高级能力
- 确保应用具备运行所需的普通权限和系统访问能力

### 2.2 需要做的授权类型

#### A. 高级执行能力

三选一：

- Shizuku
- ADB 初始化
- Root

这里的 `ADB 初始化` 应限定为：

- 一次性完成接入准备
- 建立后续 App 内可用能力链
- 不将长期连接电脑作为运行时前提

实现建议：

- 抽象 `AccessChecker`
- 分别实现：
  - `ShizukuAccessChecker`
  - `AdbInitializationAccessChecker`
  - `RootAccessChecker`

输出统一结果：

- 是否可用
- 原因
- 建议操作

#### B. 使用情况访问权限

用途：

- 识别当前前台应用

实现方式：

- 通过 `AppOpsManager` 或权限状态检查判断是否已授权
- 未授权时跳转到使用情况访问设置页

#### C. 通知权限

用途：

- 前台服务通知
- 错误提示和状态提醒

#### D. 电池优化白名单

用途：

- 提高存活率

实现方式：

- 检查是否已忽略电池优化
- 未忽略时引导到对应设置页

### 2.3 首启授权流程

建议流程：

1. 检测高级权限能力
2. 检测使用情况访问
3. 检测通知权限
4. 检测电池优化状态
5. 只要高级权限或使用情况访问不满足，就不进入完整自动化流程

## 3. 开关飞行模式 / 移动数据

### 3.1 目标

- 把“断网”和“恢复联网”做成统一动作
- 屏蔽不同执行方式的差异

### 3.2 设计方式

定义统一接口：

- `ConnectivityExecutor`

建议方法：

- `disconnect()`
- `reconnect()`
- `canExecute()`
- `getExecutionMode()`

在内部再拆：

- `AirplaneModeExecutor`
- `MobileDataExecutor`

最终组合出：

- `ShizukuAirplaneModeExecutor`
- `RootAirplaneModeExecutor`
- `ShizukuMobileDataExecutor`
- `RootMobileDataExecutor`

不建议把 `AdbExecutor` 定义成必须依赖电脑在线的长期执行器。

若保留 ADB 路线，建议定义为：

- `AdbBootstrappedExecutor`

其职责是消费 ADB 初始化后已经建立的能力，而不是要求用户常驻连接电脑。

### 3.3 执行动作前的准备

在真正切换前应做：

1. 读取当前 Wi-Fi 状态
2. 读取当前蓝牙状态
3. 读取当前目标模式
4. 检查当前执行器是否可用
5. 检查是否被 Wi-Fi 例外规则拦截

### 3.4 执行后的处理

1. 根据用户设置恢复 Wi-Fi 状态
2. 根据用户设置恢复蓝牙状态
3. 再次读取当前网络状态
4. 写入执行日志

若当前模式为“移动数据模式”且本次未改动 Wi-Fi 或蓝牙：

- 应将“保留 Wi-Fi / 蓝牙状态”记录为 no-op
- 不应为了满足配置项而额外切换 Wi-Fi 或蓝牙

### 3.5 失败处理

若动作失败：

1. 返回失败结果对象
2. 记录失败原因
3. 刷新首页与诊断页状态
4. 必要时暂停自动执行

## 4. 监控应用

### 4.1 目标

- 获取当前前台应用
- 在前台应用变化时触发规则重新判断

### 4.2 推荐实现

核心能力：

- `UsageStatsManager`

建议做法：

- 在前台服务内周期性轮询最近前台应用
- 比较当前包名和上一次包名
- 仅在发生变化时触发规则评估

### 4.3 为什么不直接依赖单一广播

因为普通应用没有一个稳定、公开、低成本的“前台应用变化广播”可以直接用，所以更现实的是：

- 轮询最近使用记录
- 配合前台服务

### 4.4 轮询建议

建议不要过于激进。

可以按状态动态调整：

- 亮屏时较高频率
- 熄屏时较低频率
- 自动化关闭时最低频率或暂停

### 4.5 应用命中判断

拿到前台应用包名后，判断顺序建议：

1. 先判断是否在黑名单
2. 若在黑名单，则直接视为非目标应用
3. 若不在黑名单，再判断是否在联网列表
4. 若在联网列表，则视为目标应用
5. 若不在联网列表，则视为非目标应用

## 5. 监控息屏 / 亮屏

### 5.1 目标

- 在屏幕熄灭后延迟断网
- 在需要时取消倒计时或恢复联网

### 5.2 推荐实现

使用系统广播监听：

- 屏幕关闭
- 屏幕点亮
- 用户解锁

广播到达后不要直接执行复杂逻辑，建议只做：

1. 更新运行时状态
2. 创建或取消倒计时任务
3. 请求规则引擎重新评估

### 5.3 倒计时实现

不要把倒计时只放在内存里。

建议：

- 内存里有当前计时任务
- 同时把关键时间点写入持久化状态

这样即使进程被杀，也能在恢复时继续判断是否应执行断网。

## 6. 监控网络环境

### 6.1 目标

- 判断当前是否处于 Wi-Fi 环境
- 获取当前 Wi-Fi、蓝牙状态
- 为例外规则和状态保留提供依据

### 6.2 推荐实现

建议拆成两个观察器：

- `NetworkStateObserver`
- `RadioStateObserver`

前者负责：

- 当前是否连上 Wi-Fi
- 当前网络类型

后者负责：

- Wi-Fi 开关状态
- 蓝牙开关状态

### 6.3 Wi-Fi 环境定义

应以“当前设备已连接可用 Wi-Fi”为准，不应只看：

- Wi-Fi 开关是否打开
- 是否保存过 Wi-Fi 网络

相关规则命名建议统一为：

- 在 Wi-Fi 环境下不自动执行恢复联网动作
- 在 Wi-Fi 环境下不自动执行断网动作

## 7. 应用扫描与应用库

### 7.1 目标

- 建立本地应用索引
- 自动识别需要联网的应用
- 支持联网列表、白名单、黑名单管理

### 7.2 扫描实现

扫描已安装应用时建议收集：

- 应用名
- 包名
- 图标引用
- 是否系统应用
- 是否有 Launcher 入口
- 是否声明 `INTERNET` 权限

### 7.3 自动识别规则

第一版建议使用启发式：

- 声明了 `INTERNET`
- 有 Launcher 入口
- 优先视作用户应用

### 7.4 列表模型

用户可见状态只有两类：

- 联网
- 非联网

列表项可额外带来源标签：

- 自动
- 手动

内部只维护 3 个列表：

- 联网列表
- 白名单
- 黑名单

其中：

- 联网列表是运行时唯一判断依据
- 白名单表示用户手动设为联网，同时应加入联网列表
- 黑名单表示用户手动设为非联网，同时应从联网列表移出

来源映射建议：

- `联网 + 自动`：在联网列表中，且不在白名单和黑名单中
- `联网 + 手动`：在白名单中，同时在联网列表中
- `非联网 + 手动`：在黑名单中，且不在联网列表中
- `非联网`：不在任何列表中

### 7.5 安装 / 卸载更新

通过系统广播监听：

- 安装
- 卸载
- 替换更新

收到事件后：

1. 增量刷新单个应用记录
2. 更新自动识别结果
3. 同步联网列表中的自动项
4. 保留已有白黑名单状态

## 8. 规则引擎

### 8.1 目标

- 把所有状态输入转换成明确动作

### 8.2 输入

- 当前前台应用
- 屏幕状态
- 是否连接 Wi-Fi
- 当前 Wi-Fi 状态
- 当前蓝牙状态
- 联网列表
- 白名单
- 黑名单
- 当前用户设置
- 当前执行器状态

### 8.3 输出

建议输出为统一动作对象：

- `DoNothing`
- `ScheduleDisconnect`
- `CancelScheduledDisconnect`
- `ReconnectNow`
- `DisconnectNow`
- `PauseAutomation`

同时建议输出统一解释字段：

- 当前统一网络状态
- 命中的关键规则
- 是否发生执行器回退

### 8.4 实现方式

不要把逻辑分散在页面、服务、广播里。

建议：

- 所有决策都集中在 `RuleEngine`
- 页面只改配置
- 服务只收集状态并调用引擎
- 执行器只负责执行

### 8.5 规则优先级

建议顺序：

1. 自动化总开关关闭
2. 高级执行器不可用
3. Wi-Fi 环境例外规则
4. 黑名单拦截
5. 联网列表命中
6. 息屏超时规则

### 8.6 亮屏 / 解锁规则

建议第一版明确为：

- 亮屏和解锁默认只更新上下文，不直接恢复联网
- 恢复联网仍以目标应用命中为准
- 若未来要支持“亮屏即恢复联网”，应作为单独开关实现

## 9. 保活与恢复

### 9.1 目标

- 在后台持续运行
- 被系统杀掉后尽快恢复

### 9.2 推荐实现

核心：

- `ForegroundService`

补偿：

- 开机恢复
- 应用更新后恢复
- 持久化运行态

明确策略建议：

- 自动化开启时启动前台服务
- 自动化关闭时停止前台服务
- 开机后仅在用户上次处于“自动化开启”状态时恢复前台服务

### 9.3 需要持久化的运行态

- 自动化是否开启
- 当前前台应用
- 是否处于息屏倒计时
- 是否处于离开应用倒计时
- 最近一次动作
- 当前执行模式

### 9.4 恢复逻辑

服务重启后：

1. 读取持久化运行态
2. 重新注册监听器
3. 重新检测高级执行器
4. 重新评估规则

若检测到首选执行器失效：

1. 尝试按优先级回退
2. 写入日志
3. 刷新首页与诊断页状态

## 10. 日志与诊断

### 10.1 目标

- 让行为可追踪
- 让故障可定位

### 10.2 日志内容

每次关键动作记录：

- 时间
- 前台应用
- 屏幕状态
- Wi-Fi 状态
- 蓝牙状态
- 命中的规则
- 动作
- 执行器
- 成功或失败
- 错误信息

### 10.3 实现方式

建议分两类：

- 结构化运行日志，存入数据库
- 调试输出日志，仅用于开发

### 10.4 日志导出建议

建议：

- 默认导出 `json`
- 默认导出最近 500 条
- 包含包名、规则命中、执行器、结果、错误信息
- 不默认附带完整原始调试输出

### 10.5 诊断页数据来源

诊断页不要自己做复杂判断，建议统一从以下来源读取：

- `AccessChecker`
- `RuntimeStatusRepository`
- `ExecutionLogRepository`

## 11. 设置与主题

### 11.1 目标

- 保存所有规则项
- 保持设置改动即时生效

### 11.2 推荐实现

使用 `DataStore` 存储：

- 总开关
- 联网模式
- 首选执行器
- 延迟秒数
- 联网列表维护策略
- Wi-Fi 例外规则
- 保留 Wi-Fi 状态
- 保留蓝牙状态
- 亮屏后不自动恢复联网
- 解锁后不自动恢复联网
- 主题模式
- 主题配色

### 11.3 设置变更后的行为

设置一旦变更：

1. 立即写入存储
2. 发出配置变更事件
3. 运行时服务重新评估当前状态

## 12. 数据层建议

### 12.1 Room 适合存

- 应用表
- 联网列表状态
- 白名单 / 黑名单状态
- 自动识别结果
- 运行日志

### 12.2 DataStore 适合存

- 用户配置
- 主题偏好
- 轻量运行状态

## 13. 推荐的代码结构

建议逐步形成以下结构：

- `access`
- `appscan`
- `apps`
- `data`
- `diagnostics`
- `executor`
- `logs`
- `rules`
- `runtime`
- `settings`
- `theme`
- `ui`

## 14. 第一阶段实现顺序

建议先做最小闭环：

1. 首启授权与接入检查
2. 应用扫描与应用列表
3. 设置存储
4. 前台应用监控
5. 息屏监控
6. 规则引擎
7. 一个执行器实现
8. 日志与诊断

第一阶段不要同时做所有高级执行器，先打通一条链路，例如：

- Shizuku + 飞行模式

跑通以后再扩展：

- Root
- 移动数据模式
- 更复杂的状态保留与恢复
