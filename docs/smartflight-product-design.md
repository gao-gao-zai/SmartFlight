# SmartFlight 产品设计文档

## 1. 项目定位

- 中文名：自动飞行
- 英文名：SmartFlight
- 平台：Android
- 核心目标：根据用户的息屏/亮屏状态、当前前台应用、黑白名单规则，自动控制设备联网能力，以降低待机耗电和后台联网消耗。

SmartFlight 本质上是一个规则驱动的系统工具应用，面向具备高级权限能力的用户。

- 高级用户：使用 Shizuku、ADB、Root 等高权限方式实现静默自动化。

当前产品方向已明确为仅服务具备高级权限能力的用户，不提供普通模式或仅提醒模式。

## 2. 设计目标

- 界面简洁、现代、直观。
- 规则设置足够清晰，不让用户在多个页面来回寻找。
- 黑名单、白名单、自动扫描出的候选应用要可视化、易编辑。
- 对权限和保活状态保持透明，用户能知道为什么生效、为什么未生效。
- 支持主题色切换、深色浅色模式切换，默认风格与 Logo 保持一致。

## 3. 能力边界

### 3.1 可稳定实现的能力

- 扫描已安装应用并识别联网候选应用。
- 监听息屏/亮屏状态。
- 识别前台应用变化。
- 根据黑名单、白名单和规则判断是否应恢复联网或断网。
- 监听应用安装、卸载、更新并刷新列表。
- 提供前台服务、日志、保活状态展示。

### 3.2 需要高权限的能力

- 静默开启/关闭飞行模式。
- 静默开启/关闭移动数据。
- 在切换飞行模式或移动数据时协调 Wi-Fi / 蓝牙状态。

这些能力不应被视为普通 Android 三方应用的标准能力，产品上必须建立高级权限前置要求。

支持的执行前提：

- Shizuku
- ADB 初始化
- Root

其中 `ADB 初始化` 的定义需要明确限制为：

- 用于一次性完成接入、授权或依赖链准备
- 不将“持续连接电脑的 ADB 会话”作为产品常规运行前提
- 若后续要保留 ADB 路线，则默认理解为“ADB 初始化后配合 App 内执行链路运行”，而不是单独长期依赖电脑

## 4. 交互与页面结构

建议采用多页底部导航结构，维持清晰的信息分层。

### 4.1 首页

用途：展示当前运行状态和关键控制入口。

包含内容：

- 总开关
- 当前联网控制模式：飞行模式 / 移动数据
- 当前执行器状态：Shizuku / ADB / Root
- 当前统一网络状态
- 当前前台应用
- 当前规则是否命中
- 最近一次动作与触发原因
- 保活状态摘要

### 4.2 应用页

用途：管理自动扫描候选应用、白名单、黑名单。

包含内容：

- 候选联网应用列表
- 白名单列表
- 黑名单列表
- 搜索框
- 筛选条件：全部 / 候选 / 白名单 / 黑名单
- 手动重新扫描按钮

交互要求：

- 显示应用图标、名称、包名、当前归类状态
- 支持一键加入白名单或黑名单
- 支持从白名单或黑名单移除
- 支持仅显示用户应用

### 4.3 规则页

用途：配置自动化行为。

包含内容：

- 息屏多少秒后开启飞行模式
- 息屏后不自动执行断网动作
- 离开目标应用多少秒后开启飞行模式
- 启动目标应用时自动关闭飞行模式
- 使用移动数据开关代替飞行模式
- 开关飞行模式时是否保留 Wi-Fi 状态
- 开关飞行模式时是否保留蓝牙状态
- 使用移动数据代替飞行模式时，是否同样保留 Wi-Fi 状态
- 使用移动数据代替飞行模式时，是否同样保留蓝牙状态
- 仅响应白名单应用
- 在 Wi-Fi 环境下是否不自动执行恢复联网动作
- 在 Wi-Fi 环境下是否不自动执行断网动作

### 4.4 主题页

用途：控制外观和个性化设置。

包含内容：

- 跟随系统 / 浅色 / 深色
- 主题色选择
- 默认主题色方案与 Logo 主色一致
- 若干备选主题色

### 4.5 诊断页

用途：展示权限、保活、执行方式、失败原因。

包含内容：

- 使用情况访问权限状态
- 通知权限状态
- 前台服务运行状态
- 电池优化豁免状态
- 自启动与后台无限制引导
- Shizuku 状态
- Root 状态
- 当前 Wi-Fi 状态
- 当前蓝牙状态
- 最近动作日志

## 5. 应用扫描策略

Android 没有“这个应用一定需要联网”的稳定公开判断接口，因此第一版不应伪装成绝对准确识别，而应使用可解释的启发式策略。

建议策略：

- 扫描声明了 `INTERNET` 权限的应用。
- 优先纳入具有 Launcher 入口的用户应用。
- 允许显示系统应用，但默认隐藏。
- 首次启动时自动扫描一次。
- 用户可手动触发重新扫描。
- 应用安装、卸载、升级后做增量刷新。

扫描结果分为四类：

- 候选联网应用
- 白名单
- 黑名单
- 忽略项

分类互斥规则建议：

- 同一应用在任一时刻只能处于“普通候选 / 白名单 / 黑名单 / 忽略项”之一
- 新操作覆盖旧状态
- 手动加入白名单或黑名单时，应自动移出忽略项
- 标记为忽略项时，应自动移出白名单或黑名单

## 6. 黑名单与白名单规则

### 6.1 黑名单

定义：即使该应用被识别为需要联网，也不因它自动关闭飞行模式或开启移动数据。

适合场景：

- 不希望某些聊天、广告、社交、游戏应用影响省电规则

### 6.2 白名单

定义：即使该应用不在自动扫描候选中，也会在启动时触发恢复联网。

适合场景：

- 银行 App
- 支付 App
- 企业办公 App
- 部分权限声明不完整但实际依赖联网的应用

### 6.3 仅响应白名单

当此项启用后：

- 自动扫描出的候选应用不再作为触发依据
- 仅白名单中的应用可触发恢复联网

### 6.4 忽略项

定义：该应用不参与自动扫描候选触发，也不在默认候选视图中参与推荐。

规则：

- 忽略项与白名单、黑名单互斥
- 忽略项不会阻止用户后续手动加入白名单或黑名单
- 一旦手动加入白名单或黑名单，即自动取消忽略状态

## 7. 自动化规则设计

### 7.1 息屏规则

- 支持设置息屏后延迟 N 秒开启飞行模式
- 支持关闭该规则
- 支持在当前连接 Wi-Fi 时不自动执行断网动作

### 7.2 前台应用规则

- 当前台应用为白名单应用时，恢复联网
- 当前台应用为候选联网应用且不在黑名单中时，恢复联网
- 离开目标应用后延迟 N 秒重新断网
- 支持在当前连接 Wi-Fi 时不自动执行恢复联网动作
- 需要单独定义亮屏或解锁后是否恢复联网

### 7.3 Wi-Fi 环境例外规则

- 当“在 Wi-Fi 环境下不自动执行恢复联网动作”启用时，命中目标应用也不执行恢复联网动作
- 当“在 Wi-Fi 环境下不自动执行断网动作”启用时，息屏或离开目标应用后不执行断网动作
- 建议以设备当前已连接可用 Wi-Fi 作为“Wi-Fi 环境”的判断条件，而不是仅 Wi-Fi 开关已打开

### 7.4 保留 Wi-Fi / 蓝牙状态

- 当启用“保留 Wi-Fi 状态”时，切换飞行模式前后应尽量恢复切换前 Wi-Fi 状态
- 当启用“保留蓝牙状态”时，切换飞行模式前后应尽量恢复切换前蓝牙状态
- 当使用“移动数据代替飞行模式”时，上述两项规则默认仍然保留在配置中，但若该模式下未主动改动 Wi-Fi 或蓝牙，则应视为无动作而不是额外执行切换
- 若执行器无法稳定实现状态恢复，应在诊断页给出能力限制说明

### 7.5 亮屏 / 解锁后的恢复规则

建议加入明确配置项：

- 亮屏后不自动恢复联网
- 解锁后不自动恢复联网
- 仅启动目标应用时恢复联网

第一版默认建议：

- 仅启动目标应用时恢复联网
- 亮屏或解锁本身不直接恢复联网

### 7.6 统一网络状态模型

建议首页、日志、诊断页共用同一套状态：

- `Offline`
- `WifiOnly`
- `CellularOnly`
- `WifiAndCellular`
- `AirplaneWithWifi`

该状态仅用于展示与规则解释，不直接替代底层原始状态记录。

### 7.7 冲突处理建议

建议规则优先级：

1. 手动总开关关闭
2. 高优先级异常状态
3. 高级权限不可用
4. Wi-Fi 环境例外规则
5. 白名单触发
6. 黑名单拦截
7. 自动扫描候选应用触发
8. 息屏超时断网

## 8. 保活设计

SmartFlight 必须从第一版开始设计保活，但目标应定义为“提高持续运行概率并支持状态恢复”，而不是追求绝对不死。

建议方案：

- 使用前台服务维持核心监听逻辑
- 通过 `BOOT_COMPLETED` 恢复服务
- 使用持久化状态在进程被杀后恢复规则判断
- 使用 WorkManager 做补偿恢复，而不是高实时监听
- 提供厂商保活引导页

需要持久化的运行状态：

- 当前是否启用自动化
- 当前是否处于息屏倒计时
- 最近前台应用
- 最近一次断网或恢复联网的原因
- 当前执行方式

## 9. 执行模式设计

建议定义统一执行器接口，由不同实现负责真正的联网控制。

执行器选择策略建议明确为：

- 默认优先级：`Root > Shizuku > ADB 初始化链路`
- 允许用户在设置页手动指定首选执行器
- 若当前首选执行器失效，可按优先级自动回退
- 自动回退后应在首页和日志中明确记录

### 9.1 ShizukuExecutor

- 面向高级用户
- 借助 Shizuku 提供高权限 shell 能力

### 9.2 AdbBootstrappedExecutor

- 面向做过 ADB 初始化的用户
- 其职责应限制为“基于初始化后已建立的能力链工作”
- 不将持续连接电脑或长期维持外部 ADB 会话作为常规运行前提

### 9.3 RootExecutor

- 面向 root 用户
- 通常具备最高静默程度和较高成功率

执行器除联网控制外，还必须负责：

- 读取切换前 Wi-Fi 状态
- 读取切换前蓝牙状态
- 按用户设置决定是否恢复 Wi-Fi 状态
- 按用户设置决定是否恢复蓝牙状态

## 10. 首页状态展示补充

当自动化总开关关闭且前台服务停止后：

- 首页仍可展示最近一次运行快照
- 当前前台应用、规则命中、待执行动作等信息均视为非实时状态
- 页面应明确标注“自动化已停止，以下信息为最近一次快照”

## 11. MVP 范围

建议第一阶段仅覆盖以下范围：

- 多页界面
- 自动扫描候选联网应用
- 黑名单 / 白名单管理
- 息屏延迟断网规则
- 前台应用触发恢复联网规则
- Wi-Fi 环境例外规则
- 保留 Wi-Fi / 蓝牙状态规则
- 亮屏 / 解锁后恢复规则
- 保活页和诊断页
- ShizukuExecutor 或 RootExecutor 任选一种先落地

暂不建议首版强推：

- 企业 Device Owner 模式
- LSPosed / Xposed 方案
- 厂商私有接口大规模适配

## 12. 后续扩展方向

- 更细颗粒度的规则组合
- 按时间段生效
- 充电状态联动
- 蓝牙 / Wi-Fi / 耳机状态联动
- 更详细的联网行为统计
- 导入导出配置
- 固定多套品牌主题之外的自定义主题能力

## 13. 主题补充

第一版建议明确为：

- 默认品牌主题 1 套
- 备选固定主题 5 套
- 第一版不开放任意自定义取色
- Logo 主色方案始终作为默认主题
