## 2. 总体目标

SmartFlight 的目标是做成一个面向高级权限用户的 Android 自动联网控制工具。

核心闭环如下：

1. 检查高级权限和使用情况访问权限。
2. 扫描已安装应用，识别需要联网的应用。
3. 管理联网列表、白名单、黑名单。
4. 监听前台应用、息屏、亮屏、网络状态。
5. 根据规则判断是否需要断网或恢复联网。
6. 通过一个高级执行器执行飞行模式或移动数据控制。
7. 记录日志，并在首页和诊断页解释当前状态。

## 3. 阶段一：补齐基础架构与数据模型

### 3.1 目标

先把项目从 UI 骨架升级为可持续扩展的应用架构。

### 3.2 任务

- **整理包结构**
  - `data`
  - `domain`
  - `settings`
  - `apps`
  - `rules`
  - `runtime`
  - `executor`
  - `diagnostics`
  - `logs`
  - `permission`
  - `di`

- **建立核心模型**
  - 应用实体：应用名、包名、是否系统应用、是否有联网权限、自动识别结果、联网列表状态、白名单、黑名单、来源标签。
  - 用户设置：总开关、联网模式、执行器偏好、息屏延迟、离开应用延迟、Wi-Fi 例外、保留 Wi-Fi/蓝牙、主题设置。
  - 运行状态：当前前台应用、屏幕状态、网络状态、最近动作、当前执行器状态。
  - 日志实体：时间、触发原因、执行动作、执行器、结果、错误信息。

- **接入 Room**
  - 存储应用扫描结果。
  - 存储联网列表、白名单、黑名单状态。
  - 存储运行日志。

- **接入 DataStore**
  - 存储规则设置。
  - 存储主题设置。
  - 存储轻量运行状态。

### 3.3 验收标准

- App 能正常编译启动。
- 数据库和设置仓库可被 ViewModel 注入。
- 能读写一组测试设置和测试应用数据。

## 4. 阶段二：实现首启接入与权限检查

### 4.1 目标

完成高级权限门槛，不满足条件时不进入完整功能。

### 4.2 任务

- **实现权限检查模块**
  - 使用情况访问权限检查。
  - 通知权限检查。
  - 电池优化白名单检查。
  - Root 可用性检查。
  - Shizuku 可用性检查。
  - ADB 初始化状态预留。

- **实现首启接入页**
  - 展示 Shizuku、ADB、Root 状态。
  - 展示使用情况访问权限状态。
  - 提供“重新检测”按钮。
  - 提供跳转系统设置入口。
  - 满足条件后才允许进入首页。

- **改造导航**
  - 从当前简单 `rememberSaveable` 切页逐步迁移到 `Navigation Compose`。
  - 增加非底部导航页面：首启接入页、应用选择页、日志详情页。

### 4.3 验收标准

- 未授权使用情况访问时，App 能引导用户授权。
- 未检测到高级执行能力时，阻止进入核心功能。
- 检测通过后进入首页。

## 5. 阶段三：实现应用扫描与应用管理页

### 5.1 目标

完成联网列表、白名单、黑名单管理。

### 5.2 任务

- **实现应用扫描器**
  - 使用 `PackageManager` 扫描已安装应用。
  - 判断是否声明 `INTERNET` 权限。
  - 判断是否有 Launcher 入口。
  - 判断是否系统应用。
  - 默认自动识别需要联网的应用。

- **实现应用数据库同步**
  - 首次启动自动扫描。
  - 手动重新扫描。
  - 安装、卸载、更新后刷新。
  - 同步联网列表中的自动项。
  - 保留用户已有白名单和黑名单。

- **实现应用页 UI**
  - 搜索框。
  - 筛选标签：全部、联网、非联网、白名单、黑名单。
  - 应用列表。
  - 显示 `自动 / 手动` 标签。
  - 设为联网。
  - 设为非联网。
  - 恢复默认。

- **实现互斥规则**
  - 同一个应用不能同时是白名单和黑名单。
  - 设为联网时自动移出黑名单，并加入白名单和联网列表。
  - 设为非联网时自动移出白名单，并移出联网列表。
  - 恢复默认时移出白名单和黑名单，再根据当前扫描结果决定是否加入联网列表。

### 5.3 验收标准

- 能扫描并展示真实安装应用。
- 能搜索、筛选、修改分类。
- 修改后重启 App 数据仍保留。

## 6. 阶段四：实现规则设置页与主题页

### 6.1 目标

让用户可以配置完整自动化规则，并保存偏好。

### 6.2 任务

- **规则页分区**
  - 联网控制模式：飞行模式、移动数据。
  - 息屏规则：息屏后自动断网、息屏延迟秒数、Wi-Fi 环境下不自动断网。
  - 前台应用规则：启动目标应用恢复联网、离开目标应用后延迟断网、离开延迟秒数、Wi-Fi 环境下不自动恢复联网、亮屏后不自动恢复联网、解锁后不自动恢复联网。
  - 应用来源规则：联网列表由自动扫描和白名单共同维护。
  - 状态保留规则：保留 Wi-Fi 状态、保留蓝牙状态。

- **主题页**
  - 跟随系统。
  - 浅色。
  - 深色。
  - 默认品牌主题。
  - 5 套固定主题色。

- **设置即时生效**
  - 修改后立即写入 DataStore。
  - 运行时服务收到配置变化后重新评估。

### 6.3 验收标准

- 所有规则项可编辑、可保存。
- 重启 App 后配置不丢失。
- 主题切换全局生效。

## 7. 阶段五：实现规则引擎

### 7.1 目标

把所有状态输入转换成明确动作，避免逻辑散落在 UI 和服务里。

### 7.2 任务

- **定义输入**
  - 自动化总开关。
  - 当前前台应用。
  - 屏幕状态。
  - 是否连接可用 Wi-Fi。
  - Wi-Fi 开关状态。
  - 蓝牙状态。
  - 联网列表。
  - 白名单。
  - 黑名单。
  - 用户规则设置。
  - 执行器状态。

- **定义输出**
  - `DoNothing`
  - `ScheduleDisconnect`
  - `CancelScheduledDisconnect`
  - `ReconnectNow`
  - `DisconnectNow`
  - `PauseAutomation`

- **实现优先级**
  1. 自动化总开关关闭。
  2. 高级执行器不可用。
  3. Wi-Fi 环境例外规则。
  4. 黑名单拦截。
  5. 联网列表命中。
  6. 息屏超时规则。
  7. 其余情况保持当前状态。

- **补充解释信息**
  - 当前命中规则。
  - 为什么执行。
  - 为什么跳过。
  - 下一步动作。
  - 是否发生执行器回退。

### 7.3 验收标准

- 规则引擎可以单元测试。
- 给定输入状态，可以稳定输出预期动作。
- 首页能展示规则命中原因。

## 8. 阶段六：实现运行时服务与系统监听

### 8.1 目标

让自动化逻辑在后台持续运行。

### 8.2 任务

- **实现前台服务**
  - 自动化开启时启动。
  - 自动化关闭时停止。
  - 持续维护运行状态。
  - 提供常驻通知。

- **前台应用监控**
  - 使用 `UsageStatsManager`。
  - 在前台服务中周期性轮询。
  - 仅在前台应用变化时触发规则评估。

- **屏幕状态监听**
  - 监听息屏。
  - 监听亮屏。
  - 监听解锁。
  - 处理息屏倒计时。
  - 亮屏或解锁时取消不适用倒计时。

- **网络状态监听**
  - 使用 `ConnectivityManager` 判断是否已连接可用 Wi-Fi。
  - 读取 Wi-Fi 状态。
  - 读取蓝牙状态。

- **保活与恢复**
  - 开机恢复。
  - 应用更新后恢复。
  - 保存当前运行态。
  - 服务重启后重新评估规则。

### 8.3 验收标准

- 开启自动化后前台服务启动。
- 息屏、亮屏、切换应用时状态能更新。
- 服务重启后能恢复最近状态。

## 9. 阶段七：实现一个执行器，先打通最小闭环

### 9.1 目标

不要一开始同时做 Root、Shizuku、ADB，先选一条链路跑通。

第一版建议优先实现：

- Shizuku + 飞行模式

之后再扩展：

- Root 执行器。
- 移动数据模式。
- ADB 初始化链路。
- 更完整的 Wi-Fi 和蓝牙恢复。

### 9.2 任务

- **定义统一接口**
  - `ConnectivityExecutor`
  - `canExecute()`
  - `disconnect()`
  - `reconnect()`
  - `getExecutionMode()`

- **实现状态保留协调器**
  - 执行前读取 Wi-Fi 状态。
  - 执行前读取蓝牙状态。
  - 执行动作。
  - 按配置恢复 Wi-Fi。
  - 按配置恢复蓝牙。
  - 失败时记录部分成功。

- **执行失败处理**
  - 返回失败对象。
  - 写入日志。
  - 首页展示异常。
  - 诊断页展示原因。
  - 必要时暂停自动化。

### 9.3 验收标准

- 满足高级权限后，能实际执行断网或恢复联网。
- 执行结果能写入日志。
- 执行失败能被首页和诊断页解释。

## 12. 建议开发顺序

1. **架构和存储**
   - 先做 `Room + DataStore + Repository + Hilt`。

2. **权限接入页**
   - 把高级权限和使用情况访问权限作为硬门槛。

3. **应用扫描和应用页**
   - 这是规则引擎的重要输入，应优先完成。

4. **规则设置页**
   - 让所有配置项先能持久化。

5. **规则引擎**
   - 集中实现所有判断逻辑，并补充单元测试。

6. **运行时服务**
   - 接入前台应用、息屏、亮屏、Wi-Fi 状态监听。

7. **执行器**
   - 先打通一条执行链路，比如 `Shizuku + 飞行模式`。

8. **日志和诊断**
   - 最后补齐可解释性和排障能力。

## 13. 风险点

- **高级权限不稳定**
  - 飞行模式和移动数据控制不是普通三方应用稳定能力，必须依赖 Shizuku、Root 或初始化后的高级链路。

- **前台应用识别有延迟**
  - `UsageStatsManager` 通常需要轮询，不能期待实时广播。

- **后台保活受系统限制**
  - 前台服务、开机恢复、电池优化引导只能提高存活率，不能保证绝对不被杀。

- **Wi-Fi 和蓝牙恢复可能失败**
  - 不同 Android 版本和厂商系统限制不同，需要日志和诊断明确提示。

- **规则冲突需要集中处理**
  - 不要在页面、广播、服务里各自判断，必须统一放进 `RuleEngine`。

## 15. 结论

建议按以下路线推进：

1. 数据层。
2. 权限门槛。
3. 应用扫描。
4. 规则配置。
5. 规则引擎。
6. 运行时服务。
7. 单执行器。
8. 日志诊断。

这个项目的关键不是先把 UI 做漂亮，而是先打通权限、规则、监听、执行、日志这一条最小闭环。完成 MVP 后，再扩展 Root、ADB、移动数据模式和更多主题能力。
