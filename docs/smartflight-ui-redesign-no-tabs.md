# SmartFlight 无 Tab 重新设计文档

> 本文档只描述画面与用户视角的操作逻辑，不包含代码实现方案。

## 1. 重新设计目标

当前界面的问题不是“不够花”，而是“信息没有主次、颜色没有语义、页面入口过多”。SmartFlight 是高权限系统工具，界面应该像一个稳定、克制的控制台，而不是多彩的功能展示页。

新设计目标：

- 完全去掉底部 Tab、顶部 Tab、分段 Tab。
- 首页只保留一个核心画面：控制台。
- 颜色只用于表达状态和主要操作，不再用多种强调色装饰不同卡片。
- 每屏只回答一个问题：现在安全吗、为什么这样、我下一步能做什么。
- 所有高风险动作必须可理解、可确认、可追踪。

## 2. 整体视觉方向

### 2.1 同类应用给我们的启发

SmartFlight 不是内容消费应用，也不是普通设置页。它更接近 Shizuku、App Ops、权限管理器、NetGuard、AdGuard、Greenify、MacroDroid、Tasker 这类系统工具或自动化工具。它们的界面通常有几个共性：

- 第一屏优先展示“当前是否正常工作”，而不是展示所有功能入口。
- 高级功能会被收进二级页面、抽屉、更多菜单或折叠区，避免干扰日常使用。
- 列表管理类页面使用搜索、筛选、更多操作，而不是在每一行塞满按钮。
- 规则类页面使用分组表单和自然语言摘要，帮助用户理解“现在这套规则会怎样工作”。
- 诊断类页面强调状态、原因和修复动作，不追求强视觉装饰。
- 品牌色通常只出现在开关、主按钮、顶部识别和选中状态，状态色另有明确含义。

SmartFlight 应采用这些工具类应用的成熟模式：默认首页是控制台，配置页是任务页面，诊断页是排错页面，而不是用 Tab 把所有模块平铺出来。

### 2.2 视觉气质

关键词：

- 冷静
- 工具化
- 清晰
- 可靠
- 接近 Logo
- 可自定义
- 少装饰

画面应像“飞行控制台的简化版”，但不要做成航空游戏 UI。用户打开应用时应感到当前状态被清楚地解释，而不是被一组彩色卡片包围。

### 2.3 Material 3 配色生成规则

默认配色必须接近 `logo.png` 的观感，而不是使用当前界面里意义不明的多色点缀。

Logo 的主要气质可以概括为：

- 深灰蓝底色
- 冷静的浅蓝灰高光
- 暖白和淡粉白的柔和背景
- 暗紫灰或褐灰作为深色对比
- 整体低饱和、偏安静，不是鲜艳科技蓝

默认主题建议命名为：

- `Logo 原色`

它不是手工写死的一套纯蓝主题，而是以 Logo 主色作为 Material 3 seed color 自动生成的完整配色。

#### 生成方式

SmartFlight 不维护多套手工色板。它使用 Material 3 的 seed color 生成完整浅色和深色配色。

默认 seed color 来自 `logo.png` 中的深灰蓝主色。Material 3 根据这个种子色生成：

- `primary`
- `secondary`
- `tertiary`
- `background`
- `surface`
- `surfaceVariant`
- `outline`
- `error`
- `onPrimary`
- `onSurface`
- 其他必要的内容色和容器色

用户看到的是“风格”，不是一张复杂的颜色表。

用户不直接编辑每一个颜色 token，也不为每张卡片单独指定颜色。

#### 基础色

在默认 `Logo 原色` 风格下，Material 3 生成后的观感应接近：

- 背景：暖白或淡粉白，接近 Logo 中的浅色区域，不使用刺眼纯白。
- 主卡片：浅暖白，与背景有轻微层级差。
- 次级卡片：极浅蓝灰或雾粉灰，用于日志、说明、辅助信息。
- 顶部或主状态区域：深灰蓝，接近 Logo 的沉稳底色。
- 文本主色：深石墨灰或暗紫灰。
- 文本次色：蓝灰。
- 分割线：低对比度雾灰。

#### 主色

Material 3 生成的 `primary` 只用于：

- 主按钮
- 总开关开启态
- 当前正在运行的关键状态
- 页面顶部极少量品牌识别
- 可点击入口的轻微强调

默认 seed color 方向：

- 深灰蓝
- 灰蓝
- 暗紫灰

不要再让每张卡片使用不同颜色点缀。

#### 状态色

状态色必须有明确含义：

- 绿色：已满足、运行正常、成功。
- 黄色：需要注意、建议处理、非阻塞。
- 红色：阻塞、失败、高风险。

红色优先使用 Material 3 的 `error` / `errorContainer`。绿色和黄色可以作为应用层语义色从当前主题派生，但不能随意变成装饰色。

状态色只出现在状态徽标、错误提示、确认弹窗中，不用于装饰标题、背景渐变或普通卡片。

### 2.4 用户自定义配色

SmartFlight 应允许用户自定义配色风格，但自定义不等于让界面重新变得花哨。自定义应该是受控的、可预览的、面向高级用户的外观能力。

自定义配色的核心不是编辑完整色板，而是选择一个 Material 3 seed color。系统根据 seed color 自动生成浅色和深色配色。

用户可以调整：

- 主题模式：跟随系统 / 浅色 / 深色。
- Seed color：影响主按钮、总开关、选中态、重点入口和整体色彩倾向。
- 背景倾向：使用 Material 3 生成结果，并允许在暖白 / 冷灰 / 深灰 / 黑色之间做轻量偏移。
- 卡片圆角：紧凑 / 标准 / 柔和。
- 颜色强度：克制 / 标准 / 高对比。

用户不应直接把每张卡片设成不同颜色。用户也不直接编辑 `primary`、`secondary`、`surface` 等 Material 3 token。

状态色默认不跟随 seed color 任意变化，避免用户把错误状态改成难以识别的颜色。

自定义页面必须提供实时预览。预览内容包括：

- 控制台主状态卡
- 成功状态
- 警告状态
- 失败状态
- 主按钮
- 列表项

如果用户选择的颜色导致文字对比度过低，界面应提示：

- `当前配色可读性较低，建议提高对比度`

### 2.5 禁止的视觉做法

新设计中应避免：

- 多张卡片分别使用蓝、绿、黄、红、粉色作为装饰点。
- 大面积无意义渐变背景。
- 无意义的彩色圆点。
- 同一屏出现过多按钮。
- 同一屏同时展示所有模块摘要。
- 把诊断项、规则项、日志项混在首页。
- 使用 Tab 或类似 Tab 的筛选条作为主要结构。
- 把用户自定义配色做成“每个模块一个颜色”的皮肤系统。

## 3. 无 Tab 信息架构

### 3.1 总体结构

应用只保留一个默认入口页面：

1. 控制台

从控制台进入以下二级页面：

1. 应用范围
2. 自动化规则
3. 诊断与日志
4. 外观设置

这些页面都不是 Tab。它们是从控制台卡片进入的独立页面，有明确标题和返回按钮。

### 3.2 导航方式

用户路径：

- 打开 App：进入控制台。
- 点击“应用范围”：进入应用范围页。
- 点击“自动化规则”：进入规则页。
- 点击“诊断与日志”：进入诊断页。
- 点击左上角返回：回到控制台。

页面之间不提供平级切换栏。用户不需要在底部 Tab 中反复寻找功能，而是从控制台按任务进入。

### 3.3 首页只做三件事

控制台只回答：

1. 当前自动化是否运行？
2. 当前网络控制状态是什么，为什么？
3. 用户下一步最可能要去哪？

其他内容全部下沉到二级页面。

## 4. 首启接入页

### 4.1 画面描述

首启接入页是用户未满足运行门槛时看到的唯一页面。

页面顶部：

- 标题：`SmartFlight 接入检查`
- 副标题：说明“需要一种高级执行能力 + 使用情况访问权限”。

页面主体是一张纵向检查卡，不拆成多个彩色区块。

检查卡内容按重要程度排列：

1. 使用情况访问权限
2. Shizuku
3. Root
4. ADB 初始化
5. 通知权限
6. 电池优化

每一项使用统一行样式：

- 左侧是状态图标。
- 中间是名称和一句解释。
- 右侧是状态文字。

状态只使用三种语义色：

- 已满足：绿色小徽标。
- 建议处理：黄色小徽标。
- 未满足且阻塞：红色小徽标。

页面底部固定一个主要按钮：

- 未满足时：`继续检查`
- 满足后：`进入控制台`

### 4.2 用户操作逻辑

用户打开 App 后：

1. 系统自动检测权限。
2. 如果核心门槛未满足，停留在接入页。
3. 用户点击某一项，页面展开该项的说明和操作按钮。
4. 用户完成授权后返回 App。
5. App 自动刷新状态。
6. 满足门槛后，底部按钮变为 `进入控制台`。

### 4.3 ADB 的用户表达

ADB 不应表现为“已经可运行的执行器”，除非真实执行链路已经可用。

如果只是记录用户已完成初始化，文案应表达为：

- `ADB 初始化记录已完成`
- `执行链路未接入`
- `暂不能作为自动执行方式`

这样避免用户误以为只勾选 ADB 就能自动开关飞行模式。

## 5. 控制台首页

### 5.1 画面描述

控制台是 App 的唯一首页。

页面整体从上到下分为四块：

1. 顶部状态区
2. 当前动作解释区
3. 任务入口区
4. 最近记录区

不再使用五个底部 Tab，也不再用多张彩色状态卡堆叠。

### 5.2 顶部状态区

顶部是一张主状态卡，占据首页首屏的主要位置。

默认 `Logo 原色` 下，主状态卡可以使用接近 Logo 的深灰蓝底色，文字使用暖白，少量高光使用浅蓝灰。这里是全应用最强的品牌识别区域，其他卡片不再重复使用大色块。

主状态卡显示：

- 自动化状态：运行中 / 已暂停 / 不可运行
- 当前控制模式：飞行模式 / 移动数据
- 当前执行方式：Shizuku / Root / 不可用
- 当前网络状态：已断网 / 已恢复联网 / 飞行模式中 / 未知

卡片右上角是总开关。

总开关旁边只显示一句短文案：

- 开启时：`规则正在监听`
- 关闭时：`所有自动动作已暂停`
- 不可开启时：`需要完成接入检查`

### 5.3 当前动作解释区

主状态卡下面是一张“为什么是这样”的解释卡。

解释卡只显示当前最重要的一条原因，例如：

- `屏幕已熄灭，将在 42 秒后断网`
- `白名单应用正在前台，已保持联网`
- `当前连接 Wi-Fi，已按规则跳过断网`
- `执行器不可用，自动化已暂停`

如果有待执行动作，显示倒计时。

如果没有待执行动作，显示：

- `暂无待执行动作`

解释卡底部可以有一个小按钮：

- `查看完整日志`

### 5.4 任务入口区

任务入口区只放三个大按钮卡：

1. `应用范围`
2. `自动化规则`
3. `诊断与日志`

每张入口卡只使用当前主题的中性色层级，不使用不同颜色装饰。默认主题下，它们应是暖白卡片或浅蓝灰卡片，而不是蓝、绿、黄、粉各一张。

每张卡显示：

- 标题
- 一句说明
- 一个右箭头

示例：

- 应用范围：`设置哪些应用会触发恢复联网`
- 自动化规则：`配置息屏、离开应用和 Wi-Fi 例外`
- 诊断与日志：`查看权限、执行器和最近动作`

外观设置不放在主入口区，放在页面右上角的更多菜单中，避免首页过载。

### 5.5 最近记录区

首页底部只显示最近一条动作记录。

内容：

- 时间
- 动作
- 结果
- 原因

如果没有记录：

- 显示空状态：`尚未执行任何自动动作`

用户点击最近记录，进入诊断与日志页。

### 5.6 用户操作逻辑

日常使用路径：

1. 用户打开 App。
2. 第一眼看到自动化是否运行。
3. 第二眼看到当前为什么联网或断网。
4. 如果要调整应用，点击 `应用范围`。
5. 如果要调整规则，点击 `自动化规则`。
6. 如果出错，点击 `诊断与日志`。

首页不要求用户理解所有系统细节，只给出当前结论和最短路径。

## 6. 应用范围页

### 6.1 画面描述

应用范围页用于管理白名单、黑名单、忽略项和候选应用。

页面顶部：

- 返回按钮
- 标题：`应用范围`
- 说明：`决定哪些应用会触发恢复联网`

顶部下方是搜索框。

搜索框下面不是 Tab，也不是彩色筛选标签，而是一行筛选摘要：

- `显示：全部应用`
- 右侧按钮：`筛选`

用户点击 `筛选` 后，打开底部弹层选择：

- 全部应用
- 会触发联网的应用
- 白名单
- 黑名单
- 已忽略
- 系统应用

### 6.2 列表项画面

每个应用是一行列表卡，不使用大卡片堆叠。

每行显示：

- 应用图标
- 应用名称
- 包名
- 当前归类状态
- 更多按钮

归类状态只用文字和小徽标：

- `候选`
- `白名单`
- `黑名单`
- `忽略`

不要在一行里同时放四个按钮。这样会让画面拥挤，也会让用户误操作。

### 6.3 用户操作逻辑

用户管理应用时：

1. 进入应用范围页。
2. 使用搜索找到应用。
3. 点击应用行或右侧更多按钮。
4. 底部弹层出现该应用的操作。
5. 用户选择：
   - 设为白名单
   - 设为黑名单
   - 设为忽略
   - 恢复为候选
6. 操作完成后列表立即更新。
7. 页面底部出现简短提示，例如：`已加入白名单`。

### 6.4 冲突处理

当用户把黑名单应用设为白名单时：

1. 弹出确认说明。
2. 文案：`该应用当前在黑名单。设为白名单后，将从黑名单移除。`
3. 操作按钮：
   - 取消
   - 设为白名单

反向操作同理。

### 6.5 空状态

未扫描时：

- 显示一张空状态卡。
- 文案：`还没有应用数据`
- 按钮：`扫描已安装应用`

扫描中：

- 显示进度说明：`正在读取已安装应用`
- 不显示空白列表。

扫描失败：

- 显示错误原因。
- 提供按钮：`重新扫描`

## 7. 自动化规则页

### 7.1 画面描述

规则页是一个纵向表单，不使用 Tab 分区。

页面顶部：

- 返回按钮
- 标题：`自动化规则`
- 说明：`设置什么时候断网，什么时候恢复联网`

页面第一块是“规则预览”。

预览用自然语言总结当前规则，例如：

`启动白名单应用时恢复联网；离开白名单应用 30 秒后断网；息屏 60 秒后断网；连接 Wi-Fi 时跳过自动断网。`

这样用户在调整开关前先理解整体行为。

### 7.2 表单分组

规则项按用户理解顺序排列：

#### A. 总行为

- 自动化总开关
- 联网控制方式
- 执行器偏好

#### B. 应用触发

- 启动目标应用时恢复联网
- 离开目标应用后断网
- 离开后延迟秒数
- 仅响应白名单应用

#### C. 息屏触发

- 息屏后自动断网
- 息屏后延迟秒数
- 亮屏后是否自动恢复联网
- 解锁后是否自动恢复联网

#### D. Wi-Fi 例外

- 连接 Wi-Fi 时不自动恢复联网
- 连接 Wi-Fi 时不自动断网

#### E. 状态保留

- 切换时保留 Wi-Fi 状态
- 切换时保留蓝牙状态

### 7.3 用户操作逻辑

用户修改规则时：

1. 进入自动化规则页。
2. 先看到自然语言规则预览。
3. 修改某个开关或延迟秒数。
4. 规则预览立即更新。
5. 页面显示 `已自动保存`。
6. 如果修改会影响当前运行状态，显示提示：`规则已更新，当前状态将在下一次检测时重新评估`。

### 7.4 高风险设置说明

以下设置旁边应有简短说明，不要只放开关：

- 使用移动数据代替飞行模式
- 保留 Wi-Fi 状态
- 保留蓝牙状态
- 亮屏后恢复联网
- 解锁后恢复联网

说明文案应解释后果，而不是技术名词。

示例：

- `开启后，SmartFlight 会尽量在切换飞行模式后恢复原来的 Wi-Fi 状态。不同系统上可能失败。`

## 8. 诊断与日志页

### 8.1 画面描述

诊断与日志页用于排错，不是首页的第二套控制台。

页面顶部：

- 返回按钮
- 标题：`诊断与日志`
- 说明：`查看为什么未执行、执行失败或权限不可用`

页面分为三块：

1. 运行条件
2. 执行器检测
3. 最近日志

### 8.2 运行条件

运行条件以列表形式展示：

- 使用情况访问
- 通知权限
- 电池优化
- 前台服务
- Shizuku
- Root
- ADB 初始化

每项只显示一个状态和一句原因。

用户点击某项后展开：

- 详细说明
- 处理按钮
- 最近检测时间

### 8.3 执行器检测

执行器检测区只放一个主要按钮：

- `重新检测执行器`

检测结果按优先级显示：

- 当前实际执行器
- 是否可执行读取
- 是否可执行写入
- 最近错误

手动切换飞行模式不直接暴露在主页面上。

如果需要保留该功能，应放在“高级操作”折叠区内。

### 8.4 高级操作

高级操作默认折叠。

展开前显示警告：

`这里的操作会直接改变设备联网状态，仅用于排障。`

高级操作包含：

- 探测飞行模式状态
- 手动开启飞行模式
- 手动关闭飞行模式
- 清空日志

每个高风险操作都需要确认弹窗。

确认弹窗必须说明：

- 即将执行什么
- 可能产生什么影响
- 是否会写入日志

### 8.5 最近日志

日志列表按时间倒序展示。

每条日志显示：

- 时间
- 动作
- 结果
- 触发原因

点击日志后进入日志详情页。

日志详情展示：

- 前台应用
- 屏幕状态
- Wi-Fi 状态
- 蓝牙状态
- 命中规则
- 执行动作
- 执行器
- 结果
- 错误详情

### 8.6 用户操作逻辑

当用户觉得“不生效”时：

1. 进入诊断与日志页。
2. 先看运行条件是否有红色阻塞项。
3. 如果没有阻塞项，点击 `重新检测执行器`。
4. 如果执行器正常，再看最近日志。
5. 点击失败日志查看原因。
6. 必要时展开高级操作进行手动探测。

## 9. 外观设置页

### 9.1 入口

外观设置不作为主页面入口，不占用首页主要空间。

入口放在控制台右上角更多菜单：

- `外观设置`

### 9.2 画面描述

外观设置页分成三块：

1. 显示模式
2. 配色风格
3. 自定义配色

页面顶部先放一张实时预览卡。预览卡展示控制台主状态卡、一个主按钮、三个状态徽标，让用户在保存前看到实际效果。

#### A. 显示模式

显示模式提供：

- 跟随系统
- 浅色
- 深色

#### B. 配色风格

默认选中：

- `Logo 原色`

该风格使用 `logo.png` 的深灰蓝作为 Material 3 seed color，由系统自动生成完整浅色和深色配色。生成后的观感应接近 Logo：深灰蓝、浅蓝灰、暖白、淡粉白、暗紫灰，整体低饱和。

可提供少量预设。每个预设本质上是一枚 seed color 加少量显示偏好，而不是一整套手工色板：

- `Logo 原色`：默认主题，使用 Logo 深灰蓝作为 seed color。
- `冷灰控制台`：使用低饱和灰蓝作为 seed color，适合浅色界面。
- `夜航深色`：使用偏深蓝灰作为 seed color，并默认使用深色模式。
- `暖白纸面`：使用暖灰粉作为 seed color，界面更柔和。
- `自定义`：用户选择一个 seed color，由 Material 3 自动生成完整配色。

不要把预设做成红、绿、紫、橙等强烈色彩套装。预设是“气质差异”，不是“换皮肤”。

#### C. 自定义配色

自定义配色放在 `自定义` 折叠区里，默认收起。

展开后允许用户设置：

- Seed color
- 背景倾向
- 卡片色倾向
- 颜色强度
- 圆角风格

用户选择 seed color 后，界面显示 Material 3 自动生成的浅色和深色预览。用户不需要理解 `primary`、`secondary`、`surface` 等 token，只需要看预览是否舒服、是否清晰。

状态色默认固定或从主题中受控派生，不跟随用户 seed color 任意变化。

如果提供状态色自定义，应放在更深一层的 `高级自定义` 中，并提示：

`状态色会影响成功、警告和失败的识别。除非你明确知道自己在改什么，否则建议保持默认。`

### 9.3 用户操作逻辑

用户进入外观设置后：

1. 先看到实时预览卡。
2. 选择显示模式。
3. 选择 `Logo 原色` 或其他预设。
4. App 根据该预设的 seed color 自动生成 Material 3 配色。
5. 预览卡立即变化。
6. 如果选择 `自定义`，打开取色器。
7. 用户选择一个 seed color。
8. App 自动生成浅色和深色预览。
9. 用户可继续调整背景倾向、颜色强度或圆角。
10. 如果配色可读性不足，页面给出提示。
11. 用户点击 `使用此风格`。
12. 返回控制台。

自定义配色保存后应出现在配色风格列表中，名称为：

- `我的风格`

如果用户再次修改，应覆盖 `我的风格`，不要无限新增主题，避免设置页变复杂。

## 10. 关键交互原则

### 10.1 一屏一个主动作

每个页面最多只有一个主按钮。

例如：

- 首启页：进入控制台
- 应用页：扫描已安装应用
- 规则页：无主按钮，自动保存
- 诊断页：重新检测执行器

### 10.2 重要信息用文字，不靠颜色

颜色只辅助理解，不能让用户必须靠颜色判断状态。

正确表达：

- `执行器不可用`
- `已跳过：当前连接 Wi-Fi`
- `失败：Shizuku 未授权`

错误表达：

- 只显示红点或黄点，没有文字说明。

### 10.3 高风险动作必须确认

以下操作必须二次确认：

- 手动开启飞行模式
- 手动关闭飞行模式
- 清空日志
- 重置 ADB 初始化状态

确认弹窗按钮文案必须具体，不使用模糊的 `确定`。

例如：

- `开启飞行模式`
- `关闭飞行模式`
- `清空全部日志`

### 10.4 说明文案要像用户语言

避免：

- `ExecutionResult.Failed`
- `ForegroundRuleDecision`
- `AppForegroundChanged`

使用：

- `执行失败`
- `检测到目标应用进入前台`
- `离开白名单应用后等待断网`

## 11. 推荐用户路径

### 11.1 第一次使用

1. 打开 App。
2. 看到接入检查页。
3. 按阻塞项完成授权。
4. 进入控制台。
5. 点击应用范围。
6. 扫描应用。
7. 添加白名单应用。
8. 返回控制台。
9. 点击自动化规则。
10. 确认息屏和离开应用规则。
11. 回到控制台开启自动化。

### 11.2 日常查看

1. 打开 App。
2. 看控制台主状态。
3. 看当前动作解释。
4. 不需要操作时直接退出。

### 11.3 出问题时

1. 打开 App。
2. 控制台看到失败或不可运行原因。
3. 点击诊断与日志。
4. 先处理红色阻塞项。
5. 再查看失败日志。
6. 必要时使用高级操作手动探测。

## 12. 新旧设计差异总结

旧设计：

- 五个底部 Tab。
- 多张彩色状态卡。
- 首页承担太多模块展示。
- 规则页和主题页占位。
- 诊断动作直接暴露。

新设计：

- 无 Tab。
- 一个控制台首页。
- 三个主要任务入口。
- 默认配色接近 `logo.png`，以 Logo 深灰蓝作为 Material 3 seed color 自动生成完整色板。
- 支持用户自定义配色，但自定义是选择 seed color 并预览生成结果，不允许回到多彩混乱状态。
- 状态色只表达语义状态。
- 规则用自然语言预览。
- 诊断动作分层，高风险操作折叠并确认。

## 13. 最终画面印象

用户打开 SmartFlight 后，应该看到一个安静的控制台：

- 顶部清楚写着自动化是否运行。
- 中间解释当前为什么联网或断网。
- 下方只有三个明确入口。
- 默认色调由 Logo seed color 生成，低饱和、灰蓝、暖白、淡粉白。
- 用户可以选择自己的 seed color 生成整体配色，但不会出现五颜六色的卡片堆叠。
- 没有 Tab。
- 没有让用户猜含义的颜色。

这个界面不追求“炫”，而追求“看一眼就知道现在发生了什么”。
